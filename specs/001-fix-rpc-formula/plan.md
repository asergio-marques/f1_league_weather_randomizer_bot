# Implementation Plan: Rpc Formula Divisor & Phase 1 Message Label

**Branch**: `001-fix-rpc-formula` | **Date**: 2026-03-04 | **Spec**: [spec.md](spec.md)
**Input**: Bug report — Phase 1 rain probability always 100%; Phase 1 message shows internal label `(Rpc)`.

## Summary

Two isolated bugs in `src/utils/`:

1. **Wrong divisor in `compute_rpc`** — `math_utils.py` divides by `3.025` instead of `3025`. This inflates every result by a factor of ~1000, causing the clamp to 1.0 to always trigger, making rain probability permanently 100%. Fix: change the literal and update the docstring.

2. **Internal label in Phase 1 message** — `message_builder.py` includes `(Rpc)` in the user-facing `**Rain Probability (Rpc)**:` line. Fix: remove the parenthetical.

Both fixes touch different files and carry no logic dependencies on each other.

## Phase 1: Design & Changes

### Data model

No data model changes. `Rpc` remains a `float` in [0.0, 1.0]; only the computation is corrected.

### Change 1 — `compute_rpc` divisor

**File**: `src/utils/math_utils.py`

```python
# Before
raw = (btrack * rand1 * rand2) / 3.025

# After
raw = (btrack * rand1 * rand2) / 3025
```

Docstring line to update:
```python
# Before
"""Compute Rpc = round((Btrack * R1 * R2) / 3.025, 2), clamped to [0.0, 1.0].

# After
"""Compute Rpc = round((Btrack * R1 * R2) / 3025, 2), clamped to [0.0, 1.0].
```

### Change 2 — Phase 1 message label

**File**: `src/utils/message_builder.py`

```python
# Before
f"**Rain Probability (Rpc)**: {pct}%\n"

# After
f"**Rain Probability**: {pct}%\n"
```

### Change 3 — Test assertion

**File**: `tests/unit/test_math_utils.py`

The test at line 30–32 asserts `compute_rpc(0.05, 1, 1) == round((0.05 * 1 * 1) / 3.025, 2)`. Both the inline comment and the expected value use the wrong divisor. Update both to `3025`.

The clamping test at line 41 (`compute_rpc(1.0, 98, 98)`) still produces a value greater than 1.0 with the correct divisor (`1.0 * 98 * 98 / 3025 ≈ 3.17`) so it continues to test the clamp correctly — no change to that assertion.

### Verification

After the fix, the production log values replay correctly:

| Input | Old result | New result |
|---|---|---|

| Btrack=0.3, R1=98, R2=98 | 1.0 (clamped) | `round(0.3 * 98 * 98 / 3025, 2)` = **0.95** |
